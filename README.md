# production-control

Веб‑приложение для управления сменными заданиями на производстве: партии, продукция, отчёты, аналитика, webhooks и фоновые задачи.

Стек: **FastAPI**, **SQLAlchemy 2 (async)**, **PostgreSQL**, **Alembic**, **Celery + RabbitMQ**, **Redis**, **MinIO**, **Docker Compose**.

---

## Основные возможности

- **Партии (Batch):**
  - создание партий пачкой из внешней системы;
  - список, просмотр, обновление статуса (закрытие);
  - агрегация продукции (синхронно и асинхронно через Celery).
- **Продукция (Product):**
  - массовое добавление по уникальным кодам;
  - признак агрегации и время агрегации.
- **Отчёты и файлы (MinIO):**
  - отчёты по партиям (Excel/PDF);
  - экспорт/импорт партий (Excel/CSV);
  - хранение файлов в бакетах `reports`, `exports`, `imports`.
- **Webhooks:**
  - CRUD подписок, HMAC‑подпись;
  - отправка через Celery, retry с экспоненциальной задержкой;
  - история доставок и 6 типов событий.
- **Аналитика:**
  - дашборд по всем партиям;
  - статистика по конкретной партии;
  - сравнение нескольких партий.

## Быстрый старт (Docker, рекомендуемый способ)

Требуется установленный **Docker** и **Docker Compose**.

1. **Собрать и запустить контейнеры:**

   ```bash
   docker compose up -d --build
   ```

2. **Выполнить миграции БД** (один раз или после изменения схемы):

   ```bash
   docker compose run --rm api alembic upgrade head
   ```

3. **Инициализировать бакеты MinIO** (один раз):

   ```bash
   docker compose run --rm api python -m scripts.init_minio
   ```

4. **Проверить, что API живой:**

   ```bash
   curl http://localhost:8000/health
   ```

   Ожидается ответ вида:

   ```json
   {"status": "ok"}
   ```

## Мини‑чек‑лист для проверки

### 1. Инфраструктура

- [ ] `curl http://localhost:8000/health` возвращает `{"status":"ok"}`.
- [ ] `http://localhost:8000/docs` открывается и показывает список эндпоинтов.
- [ ] В `http://localhost:5555` виден хотя бы один Celery worker.
- [ ] В `http://localhost:15672` (admin/admin) видно очереди Celery.
- [ ] В `http://localhost:9001` (minioadmin/minioadmin) существуют бакеты `reports`, `exports`, `imports`.

### 2. Базовый сценарий по партиям

- [ ] **Создать партию** через `POST /api/v1/batches` (тело — массив). Пример одного элемента:

  ```json
  [{
    "СтатусЗакрытия": false,
    "ПредставлениеЗаданияНаСмену": "Задание на смену 1",
    "РабочийЦентр": "РЦ-01",
    "Смена": "1",
    "Бригада": "A",
    "НомерПартии": 1,
    "ДатаПартии": "2025-02-19",
    "Номенклатура": "Изделие",
    "КодЕКН": "EKN001",
    "ИдентификаторРЦ": "rc-01",
    "ДатаВремяНачалаСмены": "2025-02-19T08:00:00",
    "ДатаВремяОкончанияСмены": "2025-02-19T20:00:00"
  }]
  ```

- [ ] **Проверить список партий** `GET /api/v1/batches` — партия видна.
- [ ] **Получить партию по ID** `GET /api/v1/batches/{id}` — возвращаются детали.

### 3. Продукция, агрегация и отчёты

- [ ] Добавить продукцию в партию: `POST /api/v1/products` с `{"batch_id": <id>, "unique_codes": ["code1", "code2"]}`.
- [ ] Запустить агрегацию: `POST /api/v1/batches/{id}/aggregate-async`, по `task_id` посмотреть статус через `GET /api/v1/tasks/{task_id}` и во Flower.
- [ ] Сгенерировать отчёт: `POST /api/v1/batches/{id}/reports` (например, `{"format": "excel"}`) и убедиться, что в MinIO в бакете `reports` появился файл.

### 4. Аналитика и webhooks (по желанию)

- [ ] Проверить `GET /api/v1/analytics/dashboard` и `GET /api/v1/analytics/batches/{id}/statistics`.
- [ ] Создать webhook подписку через `POST /api/v1/webhooks` и убедиться, что при создании партии появляются записи в `GET /api/v1/webhooks/{id}/deliveries`.
